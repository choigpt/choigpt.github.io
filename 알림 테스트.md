  1. HikariCP Pending 275 (커넥션 풀 일시 포화)                                                                                                                                                                   
                                                                                                                                                                                                                
  원인:
  - Extreme/Spike 구간(1500 VU)에서 write 작업(mark, delete, markAll)이 동시 폭주
  - markAsRead, deleteByIdAndUserId는 각각 @Transactional — 커넥션 1개씩 점유
  - 특히 deleteByIdAndUserId가 2회 UPDATE (unread 삭제 시도 → 실패 시 전체 삭제), 커넥션 점유 시간 2배
  - ec2 프로필 maximum-pool-size: 300이지만 1500 VU의 write가 동시에 몰리면 부족

  개선 방안:
  1. delete 쿼리 통합: 2단계 DELETE를 1단계로 변경 — DELETE ... WHERE id=:id AND user_id=:userId 후 SELECT is_read 체크를 사전에 하거나, unread 카운트 감소를 별도로 처리
  2. Read/Write 풀 분리: 읽기 전용 쿼리(list, unread, deepPage)는 readDataSource로 라우팅하여 write 풀과 경합 제거 — 이미 DataSourceRoutingConfig가 있으므로 @Transactional(readOnly=true) 설정 확인
  3. 비동기 write 배치: mark/delete를 즉시 DB 반영 대신 Redis에 큐잉 → 배치 flush (like 좋아요 스트림처럼)

  4. MySQL Running Threads 330, Slow Query 613

  원인:
  - countUnreadByUserId 쿼리가 서브쿼리 + COUNT(*):
  SELECT COUNT(*) FROM notification
  WHERE user_id=:userId AND is_read=false
  AND notification_id > (SELECT read_all_upto_id FROM user_notification_state ...)
  - 유저당 알림 ~200건, 인덱스 (user_id, is_read, notification_id) 있지만 서브쿼리 + 범위 스캔 비용
  - markAsReadByIdAndUserId와 deleteByIdAndUserId는 row-level lock 발생 — 같은 유저의 동시 mark/delete가 lock 경합
  - 1500 VU에서 write 비율이 높아지면 InnoDB row lock waits 32건, lock time 2.7s

  개선 방안:
  1. unread count Redis 캐시: 이미 NotificationUnreadCounter가 있으므로, countUnreadByUserId에서 DB 조회 대신 Redis 캐시 우선 — DB 쿼리 50% 이상 제거
  2. Covering Index 추가: (user_id, is_read, notification_id, content) — list 쿼리에서 테이블 접근 제거
  3. mark/delete 낙관적 처리: WHERE n.id = :id AND n.user.userId = :userId에서 영향 행 0이면 무시 (현재도 그렇지만, delete의 2단계 쿼리를 1단계로)

  4. Read p95 높음 (list 949ms, unread 798ms)

  원인:
  - p50은 24ms/14ms로 빠르지만 p95가 높은 건 극한 구간(1500 VU) 때문
  - 이 구간에서 write lock으로 인한 read 대기 발생 — InnoDB의 MVCC에서도 극한 lock 경합 시 read 지연
  - findNotificationsByUserId 네이티브 쿼리에서 user_notification_state 서브쿼리 JOIN이 매 요청마다 실행
  - 인덱스 (user_id, notification_id DESC)가 있어 정렬은 빠르지만, content 컬럼 조회를 위해 테이블 랜덤 I/O 발생

  개선 방안:
  1. Read Replica 분리: @Transactional(readOnly=true) 쿼리를 read replica로 라우팅 — write lock 영향에서 완전 격리
  2. 워터마크 인라인 캐시: user_notification_state의 read_all_upto_id를 Redis에 캐싱 → 서브쿼리 제거, 단순 WHERE user_id=:userId AND notification_id < :cursor + 앱 레벨에서 isRead 판단
  3. list 결과 Redis 캐시: 첫 페이지(cursor=null)를 Redis에 30초 TTL 캐시 — 가장 빈번한 요청 DB 제거

  개선 방안 (캐시 제외)                                                                                                                                                                                           
   
  1. HikariCP Pending — delete 2단계→1단계 + Read/Write 풀 분리                                                                                                                                                   
                                                                                                                                                                                                                
  delete 쿼리 통합: 현재 2회 DELETE → 1회로
  // Before: 2번 쿼리 (unread 삭제 시도 → 실패 시 전체 삭제)
  // After: 1번 쿼리 + wasUnread 판단을 SELECT로 사전 확인

  Read/Write 풀 분리: @Transactional(readOnly=true) 쿼리가 readDataSource로 라우팅되는지 확인 → write 풀 경합에서 읽기 격리

  2. MySQL Running/SlowQuery — 서브쿼리 제거 + Covering Index

  list/unread 쿼리의 서브쿼리를 JOIN으로 변환:
  -- Before: 매번 서브쿼리
  WHERE notification_id > (SELECT read_all_upto_id FROM user_notification_state ...)

  -- After: LEFT JOIN
  LEFT JOIN user_notification_state uns ON uns.user_id = n.user_id
  WHERE n.notification_id > COALESCE(uns.read_all_upto_id, 0)

  Covering Index: (user_id, notification_id DESC, content, type, is_read, created_at) — 테이블 랜덤 I/O 제거

  3. Read p95 — Read Replica 라우팅 확인 + 쿼리 최적화

  이미 DataSourceRoutingConfig이 있으므로 readOnly 트랜잭션이 실제 read replica로 가는지 확인

  4. SSE 63.2% — timeout 증가 + Recovery 경량화

  - SSE timeout 60s → 300s
  - 미연결 유저 Recovery 스킵
