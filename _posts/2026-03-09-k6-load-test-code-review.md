## 개요

[알림](/notification-aws-ec2-load-test/), [채팅](/chat-aws-ec2-load-test/), [피드](/feed-aws-ec2-load-test/), [정산](/settlement-aws-ec2-load-test/), [검색](/search-aws-ec2-load-test/) 5개 도메인의 EC2 부하 테스트를 진행하면서, 테스트 코드 자체의 문제로 삽질한 시간이 상당했다. 시드 데이터와 k6 시나리오의 정합성이 맞지 않으면 측정 결과를 신뢰할 수 없고, 병목 원인을 오판하게 된다. 성능 최적화가 아니라 테스트 환경 디버깅에 시간을 쓰는 상황이 반복됐다.

6개 도메인(알림, 채팅, 피드, 정산, 검색 — EC2 테스트 5개 + 클럽스케줄 — 로컬 전용)의 k6 테스트 코드를 전체 리뷰하고 발견한 10가지 문제와 수정 내용을 정리한다.

---

## 심각도 높음 — 테스트 결과 왜곡

### 1. 검색: 시드 데이터와 불일치하는 지역 데이터

시드 SQL에서 생성하는 `(city, district)` 조합은 5개다: 서울/강남구, 서울/마포구, 부산/해운대구, 대구/서구, 인천/서구.

k6 테스트의 `LOCATIONS` 배열은 10개였고, 그 중 5개(서울/서구, 서울/남구, 서울/중구, 부산/사하구, 광주/남구)가 시드에 없는 조합이었다. 지역 필터 검색의 약 50%가 무조건 0건을 반환하므로, 검색 성능이 아니라 빈 결과 반환 속도를 측정한 셈이다.

**수정**: `LOCATIONS`를 시드 SQL의 ELT 공식과 일치하는 5개로 축소.

### 2. 피드: 극소수 피드에만 집중

`setup()`에서 15개 클럽(main 5 + sub 10)의 피드만 수집했다. 총 50K 클럽 중 0.03%만 커버한다. `HIGH_COMMENT_FEEDS` 최대 45개, `HOT_FEEDS` 최대 30개에 500~700 VU가 집중되어, 실제 서비스의 분산 패턴과 동떨어진 핫스팟을 만들었다. 캐시 효과를 과대평가하고 분산 쿼리 성능을 과소측정하는 결과다.

**수정**: `SUB_CLUB_IDS`를 10개 순차 → 45개 분산(i × 1,111 간격)으로 확대하여 커버리지를 넓혔다.

### 3. 알림: 숨겨진 N+1 부하

write 성능을 측정하는 Phase(writeStorm, extremeMix, spike, doubleSpike)에서 매 iteration마다 `fetchNotificationIds()`로 추가 GET 1회를 호출했다. Phase 7(1,500 VU)에서 약 80%가 이 호출을 실행하므로, 1,500 VU 중 80%(약 1,200 VU)가 매 iteration마다 추가 read 요청을 보낸다. 문서화된 부하의 거의 2배이며, write 성능 측정에 read 부하가 혼입된다.

## 심각도 중간 — 정확성/일관성

### 4. 정산: 주석과 실제 VU 불일치

| 주석 | 실제 |
|------|------|
| Phase 2: "1200 VUs peak" | 600 VUs |
| Phase 7: "1800 VUs peak" | 900 VUs |
| Phase 8: "3000 VUs" | 1,500 VUs |

`VU_SCALE` 적용 후 실제 VU가 주석의 절반이었다. 결과 리포트 해석 시 혼동을 유발한다.

**수정**: 주석의 VU 수를 실제 값과 일치하도록 수정.

### 5. 피드/검색: VU_SCALE 미적용

알림, 채팅, 클럽스케줄 테스트는 `vu()`, `dur()`, `startAfter()` 스케일링 함수를 사용하여 로컬/EC2 간 VU를 조절할 수 있었다. 피드와 검색은 VU가 하드코딩(피드 50~1,000, 검색 50~2,000)되어 있어 환경별 스케일링이 불가능했다.

| 파일 | vu() | dur() | startAfter() |
|------|------|-------|-------------|
| chat | O | O | O |
| notification | O | O | O |
| feed | X | X | X |
| finance | X | X | X |
| search | X | X | X |
| club-schedule | O | X | X |

**수정**: 검색에 `vu()` 스케일링과 `TOTAL_USERS` import를 적용.

### 6. 피드: setup 실패 시 undefined 접근

`HIGH_COMMENT_FEEDS`가 빈 배열일 때 `Math.floor(Math.random() * 0)`이 0을 반환하고, `undefined.feedId`에서 crash가 발생한다. Phase 2/4/6/8/9/10의 10곳에서 동일 패턴이 존재했다.

**수정**: 10곳에 `.length > 0` 가드를 추가.

### 7. 알림: handleSummary 미반환

다른 5개 테스트는 `return { 'stdout': summary }`를 반환하지만 알림만 `return {}`을 반환하여 k6 summary export에 결과가 포함되지 않았다.

**수정**: `return { 'stdout': summary }`로 수정.

## 심각도 낮음 — 개선 권장

### 8. 응답 바디 검증 부재 (전 도메인)

대부분 `status === 200`만 확인한다. 서버가 200 + 빈 body 또는 잘못된 구조를 반환해도 성공으로 카운트된다. 검색 테스트의 `keywordWithLocation`만 district 필터 검증이 있고 나머지는 없다.

### 9. think time이 스트레스 테스트에 치우침

`sleep(0.02~0.05s)` 패턴이 대부분이다. 실제 사용자 행동(1~5초 think time)과 괴리가 있으므로, "1,000 VU 통과"를 "동시 1,000명 처리 가능"으로 해석하면 안 된다. 병목 탐지에는 유효하지만 동시접속자 수치로 환산하려면 think time을 현실적으로 조정해야 한다.

### 10. 정산 Phase 2/3 동시 실행

Phase 3(멱등성 750 VU)이 Phase 2(결제 600 VU)와 `startTime: '35s'`로 겹친다. 의도적일 수 있으나, Phase 3의 멱등성 결과가 Phase 2의 부하 영향을 받아 순수 멱등성 성능 측정이 어렵다.

## 교훈

부하 테스트에서 시나리오 코드의 정합성이 확보되지 않으면, 최적화 방향 자체가 잘못된다. 검색 테스트에서 50%가 빈 결과를 반환하는 상태로 "p95 24초"를 개선하려 하면, 실제 검색 성능이 아니라 빈 결과 반환 속도를 최적화하게 된다. 피드 테스트에서 15개 클럽에만 집중하면 캐시 히트율이 비정상적으로 높게 나오고, 이를 근거로 캐싱 전략을 결정하면 실 서비스에서 기대한 효과가 나오지 않는다.

정산 도메인에서 @Version NULL, pending_out=0 같은 시드 데이터 버그를 수정하는 데 실제 최적화보다 더 많은 시간을 소모했고, 검색 도메인의 이중 인코딩 문제는 mysql CLI에서는 정상으로 보여서 원인 추적에 시간이 걸렸다. 부하 테스트 코드를 짜는 것도 시드 데이터와 정확히 맞는 파라미터를 생성해야 하므로 단순하지 않다.

> **부하 테스트 코드는 애플리케이션 코드와 동일한 수준으로 리뷰해야 한다.** 시드 데이터와의 정합성, 측정 대상의 순수성(write 테스트에 read 혼입 여부), 스케일링 유연성이 확보되지 않으면 테스트 결과를 신뢰할 수 없고, 그 결과에 기반한 최적화는 방향을 잃는다.

## 시드 데이터 품질 — 반복된 패턴

6개 도메인 테스트에서 시드 데이터 문제가 반복적으로 발생했다.

| 도메인 | 이슈 |
|--------|------|
| 정산 | `@Version` NULL, `pending_out=0` 초기값 누락 |
| 피드 | 삭제된 Club을 참조하는 FK, 15개 클럽 집중 |
| 검색 | UTF-8 double encoding, location 불일치 |
| 알림 | 유저당 30,000건 비현실적 분포 |

이건 개별 실수가 아니라 테스트 환경 구축에 대한 체계 부재의 결과다. FK 정합성 검증, 데이터 분포 검증, charset 확인을 시드 생성 단계에서 자동화했어야 했다. 부하 테스트 결과의 신뢰성은 시드 데이터의 품질에 의존한다는 교훈을 얻었다.
