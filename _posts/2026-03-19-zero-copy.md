대용량 데이터를 처리할 때 성능 문제는 종종 "연산"보다 다른 곳에서 발생한다.  
특히 Java에서 파일을 읽고 처리하는 경우, I/O 이후의 처리 방식이 전체 성능에 영향을 준다.

이 글에서는 1BRC(1 Billion Row Challenge)를 예시로,  
Java에서 발생하는 I/O 병목과 zero-copy 개념을 정리한다.

---

# 문제 상황

https://github.com/gunnarmorling/1brc

1BRC(1 Billion Row Challenge)는 **10억 행의 기상 관측 데이터를 가장 빠르게 집계하는 것**을 목표로 하는 챌린지다.

### 입력 데이터

`measurements.txt` 파일에 `관측소명;온도` 형식으로 10억 행이 저장되어 있다.

```
Hamburg;12.0
Bulawayo;8.9
Palembang;38.8
St. John's;15.2
Cracow;12.6
...
```

- 약 10억 행, 수 GB 크기
- 관측소 수는 최대 10,000개
- 온도는 소수점 한 자리 (`-99.9` ~ `99.9`)

### 요구하는 출력

각 관측소별로 **최소/평균/최대 온도**를 계산하여, 관측소명 알파벳 순으로 출력한다.

```
{Abha=-23.0/18.0/59.2, Abidjan=-16.2/26.0/67.3, Accra=-10.1/26.4/66.4, ...}
```

이 정도 규모에서는 단순한 구현 방식이 그대로 성능 문제로 이어진다.

---

# 일반적인 접근

Java에서는 보통 다음과 같은 방식으로 파일을 읽는다.

BufferedReader br = new BufferedReader(new FileReader("data.txt"));  
String line;  
  
while ((line = br.readLine()) != null) {  
    process(line);  
}

이 방식은 간단하고 대부분의 상황에서 문제없이 동작한다.

하지만 데이터가 10억 행 수준이 되면, 내부적으로 발생하는 비용이 무시하기 어려워진다.

---

# 내부에서 일어나는 일

파일을 한 줄 읽을 때 실제로는 다음과 같은 흐름이 발생한다.

디스크 → 페이지 캐시 → JVM Heap → String 객체

조금 더 풀어서 보면:

1. 디스크에서 데이터를 읽어 **페이지 캐시**에 적재
    
2. 페이지 캐시의 데이터를 **JVM 힙으로 복사**
    
3. 복사된 데이터를 기반으로 **String 객체 생성**
    

이 과정이 한 줄마다 반복된다.

---

# 비용이 발생하는 지점

이 구조에서 주요 비용은 세 가지다.

### 1. 시스템 콜 반복

파일을 읽을 때마다 커널과 상호작용이 발생한다.

---

### 2. 메모리 복사

페이지 캐시에 있는 데이터를 JVM 힙으로 옮기는 과정이 추가된다.

---

### 3. 객체 생성 및 GC

각 라인이 String 객체로 생성되면서 힙에 쌓인다.

객체 생성 → Eden → Minor GC

10억 행이면 객체도 10억 개 생성된다.  
이 경우 GC가 반복적으로 발생하는 구조가 된다.

---

# 핵심 정리

디스크 I/O는 모든 방식에서 동일하게 발생한다.

차이는 그 이후에 있다.

- **페이지 캐시 → JVM 힙으로의 복사**
- **객체 생성 및 GC 비용**

---

# 페이지 캐시(Page Cache)

커널이 디스크에서 읽어온 데이터를 **RAM에 페이지(4KB) 단위로 보관해두는 캐시**다. 같은 파일을 다시 읽으면 디스크에 접근하지 않고 RAM에서 바로 반환한다.

---

# zero-copy란 무엇인가

여기서 등장하는 개념이 zero-copy다.

> zero-copy는 I/O를 없애는 기술이 아니다.
> 데이터를 **복사하지 않고 참조로 전달**하는 방식이다.

프로그래밍의 깊은 복사 / 얕은 복사와 같은 관점이다.

- `read()` = **깊은 복사**: 페이지 캐시의 데이터를 유저 공간에 새로 복사하여 별도 메모리에 보관
- `mmap()` = **얕은 복사**: 페이지 캐시의 같은 물리 메모리를 직접 참조. 데이터 이동 없음

디스크 I/O 자체는 동일하게 발생하지만, 그 이후 데이터를 전달하는 방법이 다르다.

---

# 전통적인 파일 전송 방식 (read + write)

디스크의 파일을 네트워크로 전송하는 가장 기본적인 코드를 보자.

```java
File.read(fileDesc, buf, len);
Socket.send(socket, buf, len);
```

이 단순한 코드 내부에서는 **4번의 복사**와 **4번의 컨텍스트 스위칭**이 발생한다.

버퍼는 모두 **RAM의 일부 영역**이다. OS가 용도별로 구역을 나눠놓은 것이다.

- **커널 버퍼**: RAM 중 "디스크에서 읽어온 데이터 보관용" 구역
- **소켓 버퍼**: RAM 중 "네트워크로 보낼 데이터 준비용" 구역

![read + write 전송 방식 — 4회 복사, 4회 컨텍스트 스위칭](/assets/img/posts/zero-copy/read-write.png)

| 단계 | 동작 | 복사 주체 |
|------|------|-----------|
| 1 | 디스크 → 커널 버퍼 (Read Buffer) | DMA |
| 2 | 커널 버퍼 → 사용자 버퍼 (Application Buffer) | CPU |
| 3 | 사용자 버퍼 → 소켓 버퍼 (Socket Buffer) | CPU |
| 4 | 소켓 버퍼 → NIC 버퍼 | DMA |

2단계와 3단계의 CPU 복사는 아무런 데이터 변경 없이 단순히 메모리 공간을 옮기는 작업이다. 이것이 zero-copy가 해결하려는 핵심 비효율이다.

---

# mmap

1BRC처럼 데이터를 파싱해야 하는 경우에는 mmap이 사용된다.

```c
#include <sys/mman.h>

void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset);
int munmap(void *addr, size_t length);
```

### 핵심 원리

`mmap()`의 반환값은 `void*` 포인터다. 이 포인터가 페이지 캐시의 물리 메모리를 직접 가리키므로, **포인터 역참조 = 파일 데이터 접근**이 된다. `read()`처럼 버퍼에 복사하는 것이 아니라, 파일 데이터를 가리키는 포인터를 받는 것이다.

디스크 → 페이지 캐시 ← 프로그램이 포인터로 직접 접근

- 읽기: 페이지 캐시 데이터를 직접 접근 (CPU 복사 없음)
- 쓰기: 커널 페이지 캐시를 직접 수정 → writeback 메커니즘으로 디스크 반영

Java는 포인터를 직접 다루지 않지만, NIO의 `FileChannel.map()`이 내부적으로 mmap을 사용하고 `MappedByteBuffer`를 반환하는 방식으로 활용할 수 있다.

### mmap() + write() 네트워크 전송 과정

네트워크로 전송할 때는 `write()` 시스템 콜과 함께 사용해야 한다.

![mmap + write 전송 방식](/assets/img/posts/zero-copy/mmap.png)

1. `mmap()` 호출 → 커널 모드 진입, 페이지 캐시를 사용자 공간에 매핑 (컨텍스트 스위칭 1회)
2. 매핑 주소 첫 접근 시 **페이지 폴트** 발생 → DMA가 디스크에서 페이지 캐시로 데이터 적재 (DMA 복사 1회)
3. `mmap()` 반환 → 유저 모드 복귀 (컨텍스트 스위칭 1회)
4. `write()` 호출 → 커널 모드 진입, 페이지 캐시의 물리 메모리 주소 추출 (컨텍스트 스위칭 1회)
5. CPU가 페이지 캐시 → 소켓 버퍼로 복사 (CPU 복사 1회)
6. DMA가 소켓 버퍼 → NIC로 전송 (DMA 복사 1회, 컨텍스트 스위칭 1회)

**결과**: CPU 복사 1회, DMA 복사 2회, 컨텍스트 스위칭 4회

### mmap()의 한계

- 가상 주소 공간 할당, 페이지 테이블 수정, TLB 교체 등의 오버헤드 → **짧은 생명 주기의 파일이나 소량 데이터에는 오히려 비효율적**
- 네트워크 전송 시 소켓 버퍼로의 CPU 복사가 여전히 존재 → **완전한 zero-copy는 아님**

# sendfile

리눅스 커널 2.2에서 도입. **파일 데이터를 사용자 공간을 거치지 않고 소켓으로 직접 전송**한다.

```c
#include <sys/sendfile.h>

ssize_t sendfile(int out_fd, int in_fd, off_t *offset, size_t count);
```

| 파라미터 | 설명 |
|----------|------|
| `out_fd` | 쓰기 파일 디스크립터 (소켓) |
| `in_fd` | 읽기 파일 디스크립터 (파일) |
| `offset` | 파일에서 읽기 시작할 위치 |
| `count` | 전송할 총 바이트 수 |

### sendfile() 전송 과정

![sendfile 전송 방식 — 사용자 공간 우회](/assets/img/posts/zero-copy/sendfile.png)

1. `sendfile()` 호출 → 커널 모드 진입 (컨텍스트 스위칭 1회)
2. DMA가 디스크 → 페이지 캐시로 복사 (DMA 복사 1회)
3. CPU가 페이지 캐시 → 소켓 버퍼로 복사 (CPU 복사 1회)
4. DMA가 소켓 버퍼 → NIC로 전송 (DMA 복사 1회)
5. `sendfile()` 반환 → 유저 모드 복귀 (컨텍스트 스위칭 1회)

**결과**: CPU 복사 1회, DMA 복사 2회, 컨텍스트 스위칭 **2회**

`mmap() + write()`과 복사 횟수는 동일하지만, 시스템 콜이 하나뿐이므로 **컨텍스트 스위칭이 4회 → 2회로 감소**.

단, **데이터 가공이 불가능**하다. 파일을 그대로 전송할 때만 사용할 수 있다.

---

# sendfile() + DMA Scatter/Gather

기본 `sendfile()`에서도 페이지 캐시 → 소켓 버퍼로의 CPU 복사가 1회 남아있었다.
**Linux 커널 2.4**부터 DMA Scatter/Gather를 도입하여 이를 완전히 제거했다.

### Scatter/Gather DMA란

메모리에 흩어져 있는 여러 버퍼(페이지 캐시의 페이지들)를 하나의 전송 단위로 묶어, **소켓 버퍼를 거치지 않고 NIC로 바로 전송**하는 기술.

### 전송 과정

![sendfile + DMA Scatter/Gather — CPU 복사 0회](/assets/img/posts/zero-copy/scatter-gather.png)

1. `sendfile()` 호출 → 커널 모드 진입 (컨텍스트 스위칭 1회)
2. DMA가 디스크 → 페이지 캐시로 복사 (DMA 복사 1회)
3. 커널이 페이지 캐시의 메모리 주소와 크기를 포함하는 **디스크립터 테이블** 생성 → NIC에 전달 (CPU가 수행하지만 데이터 복사 아님)
4. NIC가 Scatter/Gather DMA로 페이지 캐시에서 직접 데이터를 수집(Gather) → NIC 버퍼로 전송 (DMA 복사 1회)
5. `sendfile()` 반환 → 유저 모드 복귀 (컨텍스트 스위칭 1회)

**결과**: CPU 복사 **0회**, DMA 복사 2회, 컨텍스트 스위칭 2회 → **진정한 Zero-copy 달성**

---

# 방식별 비교

| 방식 | CPU 복사 | DMA 복사 | 컨텍스트 스위칭 |
|------|----------|----------|-----------------|
| read() + write() | 2회 | 2회 | 4회 |
| mmap() + write() | 1회 | 2회 | 4회 |
| sendfile() | 1회 | 2회 | 2회 |
| sendfile() + Scatter/Gather | **0회** | 2회 | 2회 |

> Java NIO의 `transferTo()` 메소드는 리눅스에서 내부적으로 `sendfile()` 시스템 콜을 사용하며, 기존 방식 대비 약 65%의 전송 시간을 단축한다. (IBM 문서 기준)

---

# 선택 기준

- 파일을 그대로 전송 → sendfile
- 데이터를 읽어서 처리 → mmap

### 적용 분야

- **웹 서버의 정적 파일 서빙**: 이미지, JS, CSS 등
- **메시지 큐 시스템**: Kafka 등에서 디스크 → 소켓 직접 전송
- 파일 I/O와 네트워크 I/O가 결합된 시스템이라면 zero-copy 최적화를 고려할 수 있음

# 결론

- 디스크 데이터가 어떤 경로로 애플리케이션까지 도달하는지
- 커널 모드와 유저 모드 사이에서 어떤 비용이 발생하는지
- 그 비용을 줄이기 위해 OS가 어떤 메커니즘을 제공하는지

이 개념들은 실제로 널리 활용되고 있다. Nginx는 정적 파일 서빙 시 `sendfile()` 시스템 콜을 기본으로 사용하고, Tomcat도 NIO 커넥터의 `sendfile` 옵션을 통해 정적 리소스를 전송할 때 유저 공간 복사를 건너뛴다. Kafka는 컨슈머에게 메시지를 전달할 때 `transferTo()`(내부적으로 `sendfile()`)를 사용해 디스크에서 네트워크로 데이터를 직접 전송한다.
