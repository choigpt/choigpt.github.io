이 시리즈는 자료구조를 개념이 쌓이는 순서로 정리한다. 앞의 구조가 뒤의 구조를 구성하는 부품이 된다.

---

## 시리즈 목차

### [1편: 선형 구조](/ds-1-linear/)

메모리 모델 → 배열 → 연결 리스트 → 스택 → 큐

메모리가 1차원 배열이라는 사실에서 출발한다. "연속 배치 vs 포인터 연결"이라는 자료구조의 근본적인 갈림길, 동적 배열의 amortized O(1) 분석, 단조 스택, 원형 큐까지 다룬다.

### [2편: 해시와 트리](/ds-2-hash-tree/)

해시 테이블 → BST → AVL / 레드-블랙 트리 → B-Tree / B+Tree → 힙

1편의 배열과 연결 리스트는 탐색이 O(n)이다. 이 편에서는 탐색을 O(1) 또는 O(log n)으로 만드는 구조를 다룬다. 해시 충돌 해결, BST의 편향 문제, 균형 트리의 회전, 디스크 최적화(B-Tree), 우선순위 큐(힙)까지.

### [3편: 그래프와 특수 구조](/ds-3-graph-special/)

트라이 → 그래프 → Union-Find → 세그먼트 트리 → 펜윅 트리 → 스킵 리스트 → 블룸 필터

비선형 구조와 특수 목적 구조. 접두사 검색(트라이), 그래프 탐색(BFS/DFS), 집합 병합(Union-Find), 구간 쿼리(세그먼트 트리/펜윅 트리), 확률적 자료구조(스킵 리스트, 블룸 필터)까지.

---

## 시간 복잡도 종합 비교

**배열**은 인덱스 접근이 O(1)이지만, 탐색/삽입/삭제는 모두 O(n)이다. **동적 배열**은 끝에 삽입할 때 amortized O(1)이고, 접근은 O(1)이지만 중간 삽입/삭제와 탐색은 O(n)이다. **연결 리스트**는 삽입/삭제 위치를 이미 알고 있을 때 O(1)이지만, 접근과 탐색은 O(n)이다.

**스택**은 top에서만, **큐**는 front/rear에서만 연산하므로 삽입/삭제가 O(1)이다. **해시 테이블**은 탐색/삽입/삭제 모두 평균 O(1)이지만 최악의 경우 O(n)이다.

**BST**는 평균 O(log n)이지만 편향 시 O(n)까지 떨어진다. **AVL / 레드-블랙 트리**는 항상 O(log n)을 보장한다. **B-Tree / B+Tree**도 O(log n)이며 디스크 I/O에 최적화되어 있다.

**힙**은 삽입과 삭제가 O(log n)이고 최솟값/최댓값 확인은 O(1)이지만, 임의 탐색은 O(n)이다. **트라이**는 문자열 길이 L에 대해 탐색/삽입/삭제 모두 O(L)이다.

**세그먼트 트리**는 구간 질의와 갱신이 O(log n)이다. **펜윅 트리**는 구간 합에 특화되어 탐색/삽입이 O(log n)이다. **스킵 리스트**는 확률적으로 탐색/삽입/삭제가 기대 O(log n)이다.

**Union-Find**는 Find/Union 연산이 역 아커만 함수 α(n)으로 사실상 상수 시간이다. **블룸 필터**는 k개의 해시 함수를 사용하여 삽입과 질의가 O(k)이며, 삭제는 불가능하다.

## 어떤 자료구조를 선택할 것인가

**인덱스로 빠르게 접근**해야 하면 **배열/동적 배열**이다. **키로 빠르게 조회**해야 하면 **해시 테이블**, **정렬 순서를 유지하면서 범위 검색**까지 필요하면 **균형 BST**(Java TreeMap, C++ std::map)를 쓴다.

**문자열 접두사 기반 검색**(자동완성, 사전)에는 **트라이**, **최솟값/최댓값을 반복적으로 꺼내야** 하면 **힙**(우선순위 큐)이다. **구간 합이나 구간 최솟값을 빈번하게 질의하면서 중간에 값도 바뀌는** 상황에는 **세그먼트 트리/펜윅 트리**를 쓴다.

**그룹을 병합하고 같은 그룹인지 판별**하는 문제(크루스칼, 연결 요소)에는 **Union-Find**, **대량 데이터에서 존재 여부만 빠르게 확인**하려면 **블룸 필터**(false positive 허용), **디스크 기반 대용량 인덱스**에는 **B+Tree**(MySQL, PostgreSQL의 인덱스)가 답이다.

**삽입/삭제가 빈번하고 중간 접근이 불필요**하면 **연결 리스트**, **LIFO**면 **스택**, **FIFO**면 **큐**.

자료구조를 선택할 때 이론적 복잡도만 보면 안 된다. **캐시 효율, 상수 계수, 구현 복잡도, 메모리 사용량**을 함께 고려해야 한다. O(log n)인 균형 트리보다 O(n)인 배열 순회가 n이 작을 때 더 빠른 경우가 흔하다. 프로파일링 없이 자료구조를 바꾸는 것은 성급한 최적화다.

### 시각화 도구 모음

자료구조는 머릿속에 그림이 그려져야 이해된다. 아래 사이트들에서 각 자료구조의 동작을 직접 눈으로 확인할 수 있다.

- **[VisuAlgo](https://visualgo.net):** 가장 포괄적이다. 정렬, 트리, 그래프, 세그먼트 트리 등 거의 모든 자료구조를 단계별 애니메이션으로 제공한다. 한국어도 지원한다.
- **[USFCA Data Structure Visualizations](https://www.cs.usfca.edu/~galles/visualization/Algorithms.html):** BST, AVL, 레드-블랙 트리, B-Tree, 힙, 해시 테이블 등을 직접 값을 넣어보며 조작할 수 있다.
- **[Algorithm Visualizer](https://algorithm-visualizer.org):** 코드와 시각화가 나란히 표시된다. 코드 각 줄이 실행될 때 자료구조가 어떻게 변하는지 확인할 수 있다.
- **[Big-O Cheat Sheet](https://www.bigocheatsheet.com):** 자료구조별 시간/공간 복잡도를 한 페이지에 정리해 둔 사이트다.
