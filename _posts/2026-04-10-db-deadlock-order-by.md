## 개요

Deadlock은 개념만 보면 단순하다. 두 트랜잭션이 서로가 가진 락을 기다리면서 멈추는 상태다.

문제는 이걸 실제로 맞닥뜨렸을 때다. 로컬에서는 아무 문제 없이 잘 돌아가던 코드가 부하 테스트를 돌리는 순간 갑자기 멈췄다. 로그에는 익숙한 문장이 찍혔다.

> Deadlock found when trying to get lock

처음에는 트랜잭션 설정이 꼬였다고 생각했다. JPA 내부에서 비효율적으로 락을 잡는 줄도 의심했다. 그런데 원인을 하나씩 좁혀보니 결론은 훨씬 단순했다.

> Deadlock은 쿼리나 JPA 문제가 아니라 락 획득 순서 문제였다.

---

## 시작점: 문제 없어 보이던 코드

문제는 아주 평범한 코드에서 시작됐다.

```java
Account a1 = findByIdForUpdate(id1);
Account a2 = findByIdForUpdate(id2);
```

단일 요청 기준으로 보면 이 코드는 정상이다.

1. 첫 번째 row 조회
2. 첫 번째 row 락 획득
3. 두 번째 row 조회
4. 두 번째 row 락 획득

로컬에서만 확인하면 이상할 이유가 없다. 실제로도 잘 동작했다.

---

## 부하 테스트에서만 터진 이유

문제는 같은 API가 동시에 여러 번 실행되기 시작하면서 나타난다.

예를 들어 두 요청이 거의 동시에 들어왔다고 하자.

### Tx1

- `id1 = 1`
- `id2 = 2`
- 접근 순서: `1 → 2`

### Tx2

- `id1 = 2`
- `id2 = 1`
- 접근 순서: `2 → 1`

여기서 핵심 조건이 충족된다.

> 같은 데이터를, 서로 다른 순서로 접근하고 있다.

Deadlock은 대부분 여기서 시작한다.

---

## 실제 실행 흐름

시간 순서대로 풀어보면 더 명확하다.

1. Tx1이 `id=1` row의 락을 획득한다.
2. Tx2가 `id=2` row의 락을 획득한다.
3. Tx1이 `id=2` 락을 요청하지만, 이미 Tx2가 잡고 있어서 대기한다.
4. Tx2가 `id=1` 락을 요청하지만, 이미 Tx1이 잡고 있어서 대기한다.

결과는 이렇다.

```text
Tx1: 1을 잡고 2를 기다림
Tx2: 2를 잡고 1을 기다림
```

즉,

```text
A → B
B → A
```

순환 대기 구조가 만들어진다. 이 순간 Deadlock이다.

---

## 문제의 본질

이 상황은 복잡한 락 이론 없이도 한 줄로 설명된다.

> 락을 잡는 순서가 서로 다르면 cycle이 생길 수 있다.

이걸 더 단순하게 쓰면 이렇다.

- Tx1: `1 → 2`
- Tx2: `2 → 1`

서로 반대 방향으로 잠그고 있으니, 동시에 실행되는 순간 충돌 가능성이 생긴다.

Deadlock은 "락이 오래 잡혀서" 생기는 문제가 아니다. 더 정확히는 "락 순서가 충돌해서" 생기는 문제다.

---

## 해결 방향

그렇다면 해결도 단순하다.

> 모든 트랜잭션이 같은 순서로 락을 잡게 만들면 된다.

나는 이 문제를 쿼리 레벨에서 해결했다.

```sql
SELECT *
FROM account
WHERE id = ? OR id = ?
ORDER BY id
FOR UPDATE;
```

핵심은 `ORDER BY id`다.

---

## 무엇이 바뀌었나

수정 전에는 입력값에 따라 락 획득 순서가 달라졌다.

- 어떤 요청은 `1 → 2`
- 어떤 요청은 `2 → 1`

수정 후에는 모든 트랜잭션이 동일한 순서로 row를 잠근다.

- 모든 요청이 `1 → 2`

이제 역방향이 사라진다.

```text
가능:   A → B
불가능: B → A
```

cycle이 만들어지려면 최소한 한 번은 방향이 뒤집혀야 한다. `ORDER BY`로 그 가능성을 구조적으로 제거한 것이다.

---

## 왜 ORDER BY로 Deadlock이 사라지나

Deadlock이 발생하려면 보통 아래 두 조건이 필요하다.

1. 여러 트랜잭션이 같은 자원 집합에 접근한다.
2. 그 자원을 서로 다른 순서로 잠근다.

첫 번째 조건은 비즈니스 자체 때문에 피하기 어려운 경우가 많다. 계좌 이체처럼 두 row를 함께 다뤄야 하는 작업이 대표적이다.

반면 두 번째 조건은 제어할 수 있다. 락 획득 순서를 정렬 기준으로 고정하면 된다.

즉, `ORDER BY`는 단순한 정렬이 아니라 **락 획득 순서를 deterministic하게 만드는 장치**다.

---

## 이 문제를 보고 다시 느낀 점

이번 케이스에서 흥미로웠던 점은, 로컬에서는 전혀 드러나지 않았다는 것이다. 단일 요청에서는 항상 정상 동작했고, 코드만 읽어도 별문제가 없어 보였다. 하지만 부하 테스트에서 동시성이 붙자 구조적 문제가 바로 드러났다.

이런 유형은 성능 최적화에서도 자주 본다.

- 문제: 접근 순서가 비결정적이다
- 결과: 충돌, 대기, 비효율이 발생한다
- 해결: 순서를 deterministic하게 고정한다

결국 핵심은 같다.

> 불확실성을 줄이면 동시성 문제도 줄어든다.

---

## 핵심 요약

- Deadlock은 복잡한 JPA 문제가 아니라 락 순서 문제였다.
- 로컬에서는 단일 요청이라 드러나지 않았고, 부하 테스트에서 동시 요청이 겹치며 터졌다.
- 같은 두 row를 서로 다른 순서로 잠그면 `A → B`, `B → A` 순환 대기가 생긴다.
- `ORDER BY id FOR UPDATE`로 락 획득 순서를 강제하면 모든 트랜잭션이 같은 방향으로 잠근다.
- 결과적으로 cycle이 사라지고 Deadlock도 사라진다.

---

## 정리하며

Deadlock을 처음 맞닥뜨리면 트랜잭션 전파, 격리 수준, JPA 내부 동작부터 의심하기 쉽다. 물론 그런 요소가 영향을 줄 수도 있다. 하지만 실제 원인은 훨씬 단순한 경우가 많다.

이번 케이스의 본질은 "같은 row를 서로 다른 순서로 잠근다"는 한 문장으로 끝난다. 그리고 해결도 거기에 정확히 대응한다. `ORDER BY`로 순서를 통일해 cycle 자체를 만들 수 없게 만드는 것.

> Deadlock은 락 순서 문제이고, ORDER BY는 그 순서를 강제로 통일하는 방법이다.
