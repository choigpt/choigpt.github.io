> 이 글은 [자료구조 2편: 해시와 트리](/ds-2-hash-tree/)의 해시 테이블 섹션 심화 문서다.

## 체이닝의 본질

해시 테이블은 배열이다. 해시 함수가 키를 배열 인덱스로 변환하고, 그 인덱스에 값을 저장한다. 문제는 **서로 다른 키가 같은 인덱스에 매핑**되는 충돌(collision)이다.

체이닝(Separate Chaining)은 각 버킷(배열 슬롯)에 **연결 리스트**를 달아서 충돌을 해결한다:

```
buckets[0] → NULL
buckets[1] → [key:"dog", val:3] → [key:"cat", val:7] → NULL
buckets[2] → [key:"fox", val:1] → NULL
buckets[3] → NULL
```

`"dog"`과 `"cat"`이 같은 인덱스 1에 매핑되었지만, 연결 리스트로 연결되어 둘 다 저장된다.

## 왜 연결 리스트인가

체이닝에서 연결 리스트를 쓰는 이유:

1. **동적 크기:** 한 버킷에 몇 개가 들어올지 미리 알 수 없다. 연결 리스트는 노드 단위로 할당/해제하므로 크기 제한이 없다
2. **삽입 O(1):** 체인의 앞(head)에 노드를 추가하면 된다
3. **삭제 O(1):** 이전 노드의 포인터나 이중 연결 리스트를 사용하면 포인터 조작만으로 제거. 단, 키 기반 삭제는 해당 노드를 먼저 찾아야 하므로 체인 길이 k에 대해 O(k)이다
4. **구현 단순:** 배열보다 구현이 간단하고 예외 상황이 적다

## 체인에서의 탐색 과정

`get("cat")`을 호출하면:

```
1. hash("cat") % capacity = 1       → buckets[1]로 이동
2. buckets[1]의 연결 리스트를 순회:
   - "dog" == "cat"?  아니오 → next
   - "cat" == "cat"?  예    → value 7 반환
```

체인 길이가 k이면 탐색은 O(k)이다. 좋은 해시 함수를 쓰면 n개 원소를 m개 버킷에 고르게 분배하여 기대 체인 길이는 n/m(적재율)이 된다. 적재율을 상수(예: 0.75) 이하로 유지하면 기대 탐색 시간은 O(1)이다.

## C 구현: 버킷 내부 동작

```c
typedef struct Entry {
    char *key;
    int value;
    struct Entry *next;   /* 같은 버킷의 다음 엔트리 */
} Entry;

/* 버킷(연결 리스트)에서 키를 탐색 */
Entry *bucket_find(Entry *head, const char *key) {
    for (Entry *e = head; e != NULL; e = e->next)
        if (strcmp(e->key, key) == 0)
            return e;
    return NULL;
}

/* 버킷 앞에 삽입 (O(1)) */
Entry *bucket_insert(Entry *head, const char *key, int value) {
    Entry *e = malloc(sizeof(Entry));
    e->key = strdup(key);
    e->value = value;
    e->next = head;    /* 기존 head를 밀어냄 */
    return e;          /* 새 head 반환 */
}
/* 주의: 이 구현은 중복 키를 검사하지 않는다. 같은 키로 두 번 삽입하면
   체인에 동일 키의 노드가 2개 생긴다. 프로덕션 해시 테이블에서는
   삽입 전에 bucket_find로 키 존재 여부를 확인하고, 이미 있으면
   값을 갱신(update)해야 한다. */

/* 버킷에서 삭제 */
Entry *bucket_delete(Entry *head, const char *key) {
    Entry **pp = &head;
    while (*pp) {
        if (strcmp((*pp)->key, key) == 0) {
            Entry *del = *pp;
            *pp = del->next;
            free(del->key);
            free(del);
            return head;
        }
        pp = &(*pp)->next;
    }
    return head;
}
```

`pp`(이중 포인터) 패턴에 주목하라. Linus Torvalds가 "C를 이해하는 사람과 이해하지 못하는 사람을 구분하는 기준"이라고 말한 바로 그 패턴이다. head를 삭제하는 경우와 중간을 삭제하는 경우를 **같은 코드로** 처리할 수 있다.

## Java HashMap의 진화: 리스트 → 트리

Java 8부터 `HashMap`은 체인 길이가 **8을 초과**하면 연결 리스트를 **레드-블랙 트리**로 변환한다. 체인 길이가 **6 이하**로 줄어들면 다시 연결 리스트로 되돌린다.

### 왜 8인가

좋은 해시 함수를 쓰면 체인 길이는 포아송 분포를 따른다. 적재율 0.75일 때 체인 길이별 확률:

```
길이 0:  0.60653066
길이 1:  0.30326533
길이 2:  0.07581633
길이 3:  0.01263606
길이 4:  0.00157951
길이 5:  0.00015795
길이 6:  0.00001316
길이 7:  0.00000094
길이 8:  0.00000006  ← 천만 분의 6
```

체인 길이가 8이 되는 것은 정상적인 해시 함수에서는 사실상 일어나지 않는다. 이것이 발생했다면 **의도적인 해시 충돌 공격**(HashDoS)이거나 극히 나쁜 해시 함수를 쓰고 있다는 뜻이다.

### 왜 트리인가

연결 리스트에서 탐색은 O(k)이다. 공격자가 같은 해시값을 갖는 키를 대량으로 보내면 k = n이 되어 모든 연산이 O(n)으로 퇴화한다. 레드-블랙 트리로 전환하면 최악에도 **O(log n)**이 보장된다.

```
체이닝 구조 변화:

적재율 정상 (체인 ≤ 8):
  bucket[i] → [A] → [B] → [C] → NULL
  탐색: O(k), k는 보통 1~2

HashDoS 공격 (체인 > 8):
  bucket[i] → 레드-블랙 트리
                   [B]
                  /   \
                [A]   [C]
  탐색: O(log k), 최악에도 보장
```

### 왜 6에서 다시 리스트로?

8에서 트리로 변환하고 7에서 리스트로 되돌리면, 원소 수가 7~8 사이를 왔다 갔다 할 때 변환이 반복된다(thrashing). 8→트리, 6→리스트로 **히스테리시스(hysteresis)** 간격을 두어 이를 방지한다. 동적 배열의 확장(2배)/축소(1/4) 비대칭과 같은 원리다.

### Java 소스에서 관련 상수

```java
// java.util.HashMap 내부 상수
static final int TREEIFY_THRESHOLD = 8;     // 이 길이 초과 시 트리 변환
static final int UNTREEIFY_THRESHOLD = 6;   // 이 길이 이하 시 리스트로 복원
static final int MIN_TREEIFY_CAPACITY = 64; // 테이블 크기가 64 미만이면 트리 변환 대신 resize
```

`MIN_TREEIFY_CAPACITY = 64`도 중요하다. 테이블이 작을 때(64 미만) 체인이 길어지면 트리로 변환하는 대신 **테이블을 2배로 키운다(resize)**. 테이블이 작으면 충돌이 자연스럽게 많으므로 트리 변환보다 resize가 더 효과적이다.

## 체이닝 vs 오픈 어드레싱

**체이닝**은 각 버킷에 연결 리스트(또는 트리)를 달아서 충돌을 해결한다. 적재율이 1.0을 초과해도 동작하고, 삭제가 간단하며, Java HashMap처럼 트리 변환으로 최악 성능을 보호할 수 있다. 대신 포인터를 따라가야 하므로 캐시 효율이 나쁘고, 포인터 오버헤드로 메모리를 더 쓴다.

**오픈 어드레싱**은 충돌 시 배열 내 빈 슬롯을 탐색(probing)하여 해결한다. 포인터 없이 연속 메모리를 탐색하므로 캐시 효율이 좋고, Python dict와 Rust HashMap이 이 방식이다. 하지만 적재율 0.7~0.8을 넘으면 클러스터링이 발생하여 성능이 급격히 나빠지고, 삭제 시 tombstone을 남겨야 해서 구현이 복잡하다.

정리하면, 체이닝은 **안정적**(적재율이 높아져도 성능 저하가 점진적), 오픈 어드레싱은 **빠르다**(캐시 효율이 좋아 평균적으로 더 빠름).

## 정리: 연결 리스트가 부품인 이유

해시 테이블의 체이닝에서 연결 리스트가 쓰이는 이유를 한 문장으로 요약하면:

> **크기를 미리 알 수 없는 컬렉션에서 O(1) 삽입/삭제가 필요하고, 순회 성능은 중요하지 않을 때** 연결 리스트가 최적의 부품이다.

체이닝의 각 버킷이 정확히 이 조건에 해당한다. 버킷 크기는 해시 분포에 따라 달라지고, 삽입/삭제가 빈번하며, 한 버킷 내 순회 길이는 적재율로 제한되므로 캐시 효율이 문제되지 않는다.
