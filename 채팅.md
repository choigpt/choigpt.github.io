  2. 쿼리 최적화 — user_chat_room 조회가 매번 DB 히트. Redis 캐시로 변환 가능
  3. 읽기 전용 쿼리 분리 — 메시지 목록/커서 페이징은 HikariCP 대신 read replica 사용
  4. connection-timeout 단축 (현재 15s) — 대기 중 빨리 실패시켜 cascade 방지

  - 시작 시: min-idle=50개만 생성                                                                                                                                                                                  
  - 부하 증가 시: 요청 들어올 때마다 하나씩 생성 → max(400)까지

  이건 warm-up 문제다. 실제 운영에서는 배포 후 트래픽이 점진적으로 올라오므로 문제 안 됨. 필요하면 min-idle을 높이면(예: 200) cold start 시에도 즉시 대응 가능.
