  문제점 및 권장사항

  1. App 서버 커널 튜닝 미적용

  메모리에 기록해둔 커널 최적화가 현재 적용되어 있지 않습니다:

  ┌───────────────────────────┬───────┬──────┬──────────────────────────────────────────┐
  │           설정            │ 현재  │ 권장 │                   영향                   │
  ├───────────────────────────┼───────┼──────┼──────────────────────────────────────────┤
  │ net.core.rmem_max         │ 212KB │ 16MB │ TCP 수신 버퍼 부족 → 고부하 시 패킷 드롭 │
  ├───────────────────────────┼───────┼──────┼──────────────────────────────────────────┤
  │ net.core.wmem_max         │ 212KB │ 16MB │ TCP 송신 버퍼 부족                       │
  ├───────────────────────────┼───────┼──────┼──────────────────────────────────────────┤
  │ tcp_keepalive_time        │ 7200s │ 60s  │ 죽은 커넥션 2시간 유지 → 리소스 낭비     │

  └───────────────────────────┴───────┴──────┴──────────────────────────────────────────┘

  2. MySQL Buffer Pool 심각하게 부족

  - Buffer Pool: 3GB
  - 전체 데이터+인덱스: 32GB
  - 비율: 9.5% — InnoDB 권장 70~80%에 크게 미달
  - 디스크 I/O 과다 발생 → 쿼리 지연의 주요 원인

  3. MySQL innodb_log_file_size 너무 작음

  - 현재: 48MB (기본값)
  - 권장: 512MB~1GB (쓰기 부하 높은 환경)
  - 로그 파일이 작으면 체크포인트 빈도 증가 → 쓰기 성능 저하

  4. MySQL innodb_flush_method = fsync

  - 현재: fsync (기본값)
  - 권장: O_DIRECT — 이중 버퍼링 방지, buffer pool 효율 극대화
  - (런타임 변경 불가, my.cnf 수정 후 재시작 필요)

  5. Redis maxmemory 512MB

  - 현재: 512MB, allkeys-lru
  - 인프라 서버 여유 RAM 8.2GB → Redis에 1~2GB 할당 가능
  - 캐시 eviction이 빈번하면 cache miss → DB 부하 증가

  6. k6 서버 커널 튜닝 부족

  - net.core.rmem_max = 212KB → 응답 수신 버퍼 부족
  - tcp_tw_reuse = 2 (기본) → TIME_WAIT 소켓 재활용 비활성

  정상인 부분

  - ulimit -n: 65535 (충분)
  - ip_local_port_range: 1024~65535 (k6 포트 충분)
  - MySQL max_connections: 800 (HikariCP 300+300에 충분)
  - HikariCP R/W 분리: write-pool + read-pool 정상 동작
  - Redis pool: maxActive=128, timeout=5s (적절)
  - innodb_flush_log_at_trx_commit: 2 (성능 우선, 적절)
  - innodb_io_capacity: 4000/20000 (SSD 환경 적절)
  - innodb_lock_wait_timeout: 3s (고부하 대응 적절)
  - slow_query_log: ON, 2초 (모니터링 활성화)
  - Kafka: healthy, 정상 연결
  - Docker 컨테이너: 12개 모두 healthy/up
